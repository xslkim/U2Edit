## Python API 能力边界（POC 观察）

- **方案 A（Python）**：适合 **编辑器内批处理**、**资源创建**、**工程规范检查**、与 **Editor Utility** 组合；本仓库脚本已验证「可编程创建 Widget Blueprint 资源」这一条主路径。  
- **边界**：UMG **WidgetTree 精细布局**在不同 Unreal 5.x 小版本上的 **Python 暴露不一致**，难以在单一脚本里 **无歧义、可 CI** 地写完与 `tasks.md` 逐条等价的 Slot 操作；因此 POC 将 **完整控件树 + Slot 参数** 明确为 **本机 Designer 对照 tasks 手工补齐**（见 `poc/unreal/README.md`）。

## D1 决策结论（必填）

| 决策项 | 结论 |
|--------|------|
| D1 — Unreal 导出方案 | **已定** |
| 主路径 | **A：Python** — 用于编辑器侧自动化（资源生成、校验、批处理、与后续导出器衔接） |
| 补充/兜底 | **C：JSON + 插件** — 承载 **控件树细节与引擎差异**（避免 Python 直接操作 UMG 内部 API 的版本漂移）；若个别能力 Python 无法覆盖，再用 **B：C++** 做局部 Editor 模块 |

> 说明：上述组合在 POC 阶段视为 **「方案 A 可用」**；不是要求“纯 Python 一行不改”，而是 **以 Python 为主力、JSON 为数据真相、必要时 C++ 补洞**。

## 风险与后续

- 生产级导出应在 M4 前固定 **目标 UE 小版本**，并为 JSON schema 做 **版本号**（与 `project.yaml` / schema 迁移策略对齐）。  
- 若后续验证发现 Python 在某版本完全无法创建 WBP，再评估将「资产创建」迁移为 **C++ Factory**（仍建议保留 JSON 为中间格式）。
